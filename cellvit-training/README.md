# cellvit-training/

H&E classifier-head training workflow that fine-tunes a CellViT-SAM-H-x40
backbone using QuST-derived Xenium cell-type labels. Produces the per-tissue
TorchScript heads consumed by WSInsight's Model Zoo.

The workflow is **tissue-agnostic**: anything that varies per tissue lives
under [`trainingset/<tissue>/`](trainingset/) вЂ” specifically `label_map.yaml`
and `train_configs/<backbone>/fold_*.yaml`. The same wrappers train every
tissue.

## Layout

```
cellvit-training/
в”њв”Ђв”Ђ pipeline/                # tissue-agnostic Python + bash drivers
в”‚   в”њв”Ђв”Ђ make_splits.py       # exported tiles в†’ train/val splits
в”‚   в”њв”Ђв”Ђ train.sh             # 4-step training wrapper
в”‚   в”њв”Ђв”Ђ validate.sh          # re-run validation against a finished run
в”‚   в””в”Ђв”Ђ validate_classifier.py
в”њв”Ђв”Ђ qupath/                  # QuPath Groovy helpers (CLI-batch capable)
в”‚   в”њв”Ђв”Ђ run_qust_pipeline.groovy  # tissue detection + StarDist + Xenium cluster transfer
в”‚   в”њв”Ђв”Ђ load_mapping.groovy       # cluster_id в†’ label-name on detections
в”‚   в”њв”Ђв”Ђ reset_clusters.groovy     # revert PathClass to Xenium cluster_id (for re-apply)
в”‚   в””в”Ђв”Ђ export_tiles.groovy       # detections в†’ tile PNG + per-tile CSV
в”њв”Ђв”Ђ trainingset/             # exported tiles + per-fold training configs
в”‚   в””в”Ђв”Ђ <tissue>/
в”‚       в”њв”Ђв”Ђ label_map.yaml   # canonical int в†” label-name table
в”‚       в”њв”Ђв”Ђ train/{images,labels}/
в”‚       в”њв”Ђв”Ђ splits/fold_0/{train,val}.csv
в”‚       в””в”Ђв”Ђ train_configs/SAM-H-x40/fold_0.yaml
в”њв”Ђв”Ђ cellvit/                 # vendored CellViT-plus-plus + base weights
в”‚   в”њв”Ђв”Ђ CellViT-plus-plus/   # upstream checkout (patched locally; not committed)
в”‚   в””в”Ђв”Ђ models/CellViT-SAM-H-x40.pth
в””в”Ђв”Ђ models/                  # promoted heads (yaml side-cars tracked; *.pth gitignored)
    в””в”Ђв”Ђ <head-name>/{config.yaml, label_map.yaml, model_best.pth}
```

The QuPath project that holds the per-sample annotations (foreground,
StarDist nuclei, Xenium cluster IDs) lives at `../data/qprj/project.qpproj`.
It's the single source of truth for the H&E-anchored cell labels consumed
by `export_tiles.groovy`.

### What's tracked vs derived under `trainingset/`

`trainingset/` is **mostly git-ignored** вЂ” bulk content is regenerated by
running the pipeline against `data/xenium/`.

Tracked in git:

- `trainingset/<tissue>/label_map.yaml` вЂ” canonical int в†” label-name table
  (hand-authored; the QuST/export pipeline and the training config both
  consume this exact file).
- `trainingset/<tissue>/train_configs/<backbone>/fold_*.yaml` вЂ” authoring
  training configs (with `${CELLVIT_TRAINING_ROOT}/...` tokens). The
  wrapper resolves them via `envsubst` at run time.

Not committed (regenerated by pipeline):

- `trainingset/<tissue>/train/`  вЂ” exported PNG tiles + per-tile label CSVs
- `trainingset/<tissue>/test/`   вЂ” held-out tiles
- `trainingset/<tissue>/splits/` вЂ” per-fold train/val CSV lists
- `trainingset/<tissue>/cache/`  вЂ” HDF5 dataloader caches (multi-GB per
  tissue; rebuilt automatically on first epoch)
- `trainingset/<tissue>/train_configs/<backbone>/.fold_*.resolved.yaml` вЂ”
  `envsubst` output materialized at training time

## Path portability

Everything is anchored on script location вЂ” no path is hard-coded to
`/workspace/...`. The wrappers derive:

```
PIPELINE_DIR           = dirname(BASH_SOURCE)
CELLVIT_TRAINING_ROOT  = PIPELINE_DIR/..
PROJECT_ROOT           = CELLVIT_TRAINING_ROOT/..
```

and export `${PROJECT_ROOT}` and `${CELLVIT_TRAINING_ROOT}` so that
`envsubst` can materialize the placeholders inside the training YAML before
handing it to CellViT++. The whole tree can be moved or renamed with no
edits вЂ” set `PYTHON=<...>` to override the interpreter if needed.

## End-to-end pipeline (per tissue)

For a tissue named `<tissue>` (e.g. `pantissue`):

```bash
# 0. (one-time) curate cell-type labels with kurtorank to produce
#    celltype_assignment_pantissue_label.csv in each sample's outs/.
#    Hand-author trainingset/<tissue>/label_map.yaml (int в†” label-name).

# 1. QuST detection + Xenium cluster transfer (headless CLI batch over every
#    image in the QuPath project). Runs PetesSimpleTissueDetection +
#    StarDistCellNucleusDetection + XeniumAnnotation in one go:
QuPath script -s -p ../data/qprj/project.qpproj \
    qupath/run_qust_pipeline.groovy

# 2. Remap cluster_id в†’ pantissue label on every detection (CLI batch):
QuPath script -s -p ../data/qprj/project.qpproj \
    -a /abs/path/celltype_assignment_pantissue_label.csv \
    qupath/load_mapping.groovy

#    (To re-apply an updated celltype_assignment_*.csv without re-running
#     tissue detection + StarDist, run qupath/reset_clusters.groovy first to
#     revert PathClass back to the Xenium cluster_id, then re-run
#     load_mapping.groovy with the new CSV.)

# 3. Export tiles + per-tile cell CSVs (CLI batch):
#    Edit OUTPUT_ROOT at top of export_tiles.groovy first.
QuPath script -p ../data/qprj/project.qpproj qupath/export_tiles.groovy

# 4. Build train/val splits from the exported tiles
python pipeline/make_splits.py --tissue <tissue>

# 5. Train head + auto-validate + export TorchScript (4 steps in one)
bash pipeline/train.sh <tissue>                       # SAM-H-x40, fold_0
bash pipeline/train.sh <tissue> SAM-H-x40 fold_0 <task>   # explicit form

# 6. Re-run only validation against a finished training run
bash pipeline/validate.sh <tissue>
```

`train.sh` runs four steps in sequence: train the `LinearClassifier` head,
locate `model_best.pth` under
`cellvit/CellViT-plus-plus/logs_local/<TIMESTAMP>_<tissue>-<task>-<backbone>/checkpoints/`,
emit a confusion matrix + classification report, and convert the checkpoint
to TorchScript at 1024Г—1024.

## Adding a new tissue

1. Create `trainingset/<tissue>/label_map.yaml` (int в†” label-name, 0..N-1
   contiguous). This is the canonical mapping.
2. Add the tissue's H&E images to `../data/qprj/project.qpproj`.
3. Curate labels (headless): `qupath/run_qust_pipeline.groovy` в†’
   `qupath/load_mapping.groovy` with the sample's
   `celltype_assignment_<tissue>_label.csv`.
4. Set `OUTPUT_ROOT` at the top of `qupath/export_tiles.groovy` to
   `trainingset/<tissue>/`, then run it via the QuPath CLI.
5. Author `trainingset/<tissue>/train_configs/SAM-H-x40/fold_0.yaml`
   (copy from an existing tissue; set `num_classes`, `label_map`, and
   `weight_list` to match the tissue's class distribution).
6. `pipeline/make_splits.py --tissue <tissue>` then
   `pipeline/train.sh <tissue>`.

### Local promotion path

Training runs write checkpoints under
`cellvit/CellViT-plus-plus/logs_local/<run_id>/checkpoints/` and W&B logs
under `.../logs_local/wandb/`. Both paths are git-ignored. After
`train.sh` finishes, the chosen head is promoted to
`cellvit-training/models/<head-name>/`:

- `model_best.pth` вЂ” checkpoint (git-ignored; publish to HF Hub separately)
- `config.yaml`    вЂ” training config used to produce it (tracked)
- `label_map.yaml` вЂ” int в†” label-name mapping (tracked)

Only the small YAML side-cars are committed, so the head is reproducible
without committing the `.pth`.

# Dataset format specification

The rest of this document is the complete format spec for the CellViT++
classifier training set. **`pantissue`** (12-class, pan-tissue) is used as
the running example because that is the head currently set up in this repo.
Earlier breast / colorectal 11-class setups follow the same layout вЂ” only
`label_map.yaml`, `num_classes`, and `weight_list` differ.

## 1. Overview

**Goal:** Export 1024Г—1024 px image tiles and matching cell-detection label CSVs
from H&E whole-slide images (Xenium `morphology.ome.tif`), with cell centroids
labelled by integer class id from the Xenium cell-type assignment.

**Model to train:** `CellViT-SAM-H-x40`
**Dataset class used in training code:** `DetectionDataset`
**Framework:** CellViT-plus-plus (`cellvit/CellViT-plus-plus/`)

## 2. Input Files (per Xenium sample)

All input files are under each sample's `outs/` folder.

| File | Description |
|---|---|
| `morphology.ome.tif` | Full H&E whole-slide image (OME-TIFF). **This is the image to open in QuPath.** Do NOT use `morphology_focus/` files (those are stain channels). |
| `cells.csv.gz` | Per-cell: `cell_id`, `x_centroid` (Вµm), `y_centroid` (Вµm), plus QC columns. Coordinates are in the Xenium tissue coordinate space (microns). |
| `analysis/clustering/gene_expression_graphclust/clusters.csv` | Per-cell: `Barcode` (= `cell_id`), `Cluster` (integer 1-based). |
| `celltype_assignment_<tissue>_label.csv` | Per-cluster: `classification` (cluster int), `cell_type` (string). Used to map Cluster в†’ cell type label. For `pantissue` the file is `celltype_assignment_pantissue_label.csv`. |

### Coordinate system
- `x_centroid` and `y_centroid` in `cells.csv.gz` are in **microns** in the Xenium slide coordinate space.
- QuPath, when the slide is opened with correct pixel calibration, also works in microns.
- To convert microns to pixels at the export resolution: `pixel = micron / MPP`.

## 3. Target Export Specification

| Parameter | Value |
|---|---|
| **Export MPP** | **0.25 Вµm/pixel** (40Г— equivalent) |
| **Tile size** | **1024 Г— 1024 pixels** |
| **Tile overlap** | **64 pixels** (on each edge, i.e. export stride = 960 px = 240 Вµm) |
| **Image format** | **PNG, RGB (3-channel)** |
| **Coordinate frame for labels** | Pixel coordinates **relative to the tile's top-left corner** at 0.25 MPP |

> **Why 0.25 MPP?** The CellViT-SAM-H-x40 network was trained at 0.25 Вµm/px. The
> inference pipeline enforces this вЂ” any other resolution requires rescaling and
> degrades performance.

## 4. Cell-Type Label Mapping (`pantissue`)

### 4.1 Pan-tissue label scheme: 12 classes

This is the primary label set produced by KurtoRank's `pantissue` profile
(`celltype_assignment_pantissue_label.csv`). The integer assignment is
fixed and global вЂ” do not renumber per sample.

| Integer (`class_int`) | `cell_type` string |
|---|---|
| 0 | `lymphoid` |
| 1 | `other_stromal_mesenchymal` |
| 2 | `epithelial` |
| 3 | `tumor` |
| 4 | `myeloid` |
| 5 | `endothelial` |
| 6 | `neural` |
| 7 | `plasma` |
| 8 | `other_granulocyte` |
| 9 | `fibroblast` |
| 10 | `neutrophil` |
| 11 | `mast` |

The same mapping is the source of truth in
[`trainingset/pantissue/label_map.yaml`](trainingset/pantissue/label_map.yaml).
Per-class frequencies (used to set `weight_list:`) are computed across the
actual exported tiles and recorded as comments in
[`trainingset/pantissue/train_configs/SAM-H-x40/fold_0.yaml`](trainingset/pantissue/train_configs/SAM-H-x40/fold_0.yaml).

> Class integers must be **0-indexed** and **consecutive**. The value in the label
> CSV is always the integer, not the string.

### 4.2 Earlier breast 11-class scheme

Earlier breast-only training (now retired; head archived) used an
11-class `hne_label` scheme: `lymphocyte, malignant_epithelial,
epithelial, plasma_cell, macrophage_like, fibroblast_like, pericyte,
endothelial, basophil, adipocyte, mast_cell`. The `pantissue` scheme above
is the recommended replacement.

## 5. Label Join Pipeline (in QuPath)

Cell labels are anchored to H&E nuclei, not to Xenium morphology (DAPI)
centroids вЂ” the offset between the two modalities can be several Вµm due to
fixation, tissue deformation and z-plane mismatch (this is the central
point of the QuST paper). The join therefore happens **inside QuPath**:

1. **Foreground annotation** вЂ” QuST в†’ `PetesSimpleTissueDetection` per image.
2. **H&E nuclear detection** вЂ” QuST в†’ `StarDistCellNucleusDetection`,
   restricted to the tissue annotation. Each detection's `PathClass` is
   left unset at this stage.
3. **Xenium cluster transfer** вЂ” QuST в†’ `XeniumAnnotation`. Reads the
   sample's `outs/analysis/clustering/.../clusters.csv` and assigns each
   H&E detection the nearest Xenium cell's cluster id. After this step
   every detection carries `PathClass.name = <cluster_id>` (e.g. `"1"`,
   `"2"`, вЂ¦).

Steps 1вЂ“3 are batched headless across every image in the QuPath project by
[`qupath/run_qust_pipeline.groovy`](qupath/run_qust_pipeline.groovy) вЂ”
see the End-to-end pipeline block at the top of this README for the CLI
invocation.

4. **Cluster в†’ label remap** вЂ” run
   [`qupath/load_mapping.groovy`](qupath/load_mapping.groovy) with the
   sample's `celltype_assignment_<tissue>_label.csv` (2 columns:
   `classification, cell_type`). PathClass becomes the label string
   (`"tumor"`, `"lymphoid"`, вЂ¦).
5. **Save** the QuPath project so the relabelled detections persist into
   each image's `.qpdata` (the `-s` flag does this for the headless
   scripts above).

To re-apply an updated `celltype_assignment_*.csv` without re-running
tissue detection + StarDist, use
[`qupath/reset_clusters.groovy`](qupath/reset_clusters.groovy) first to
revert each detection's `PathClass` back to its Xenium cluster id, then
run `load_mapping.groovy` with the new CSV.

No Python pre-processing reads anything from inside the QuPath project.
The int в†” label-name mapping consumed at training time lives in
`trainingset/<tissue>/label_map.yaml` and is hand-authored.

## 7. Tile Image Format

- **Format:** PNG
- **Color:** RGB (3 channels). If the OME-TIFF has more channels, export only the H&E channel composite (or channel 0 as RGB).
- **Size:** exactly 1024 Г— 1024 pixels
- **Bit depth:** 8-bit per channel (uint8)
- **MPP at export:** 0.25 Вµm/px

### Tile naming convention

```
{sample_tag}_tile_{tile_index:04d}.png
```

The `sample_tag` is derived by `export_tiles.groovy` from the image name
in the QuPath project. It must be unique across all samples that feed
into the same tissue's training set; the tile image and its label CSV
must share the **same stem** (filename without extension).

## 8. Label CSV Format

One CSV file per tile. **No header row.** Three columns: `x, y, class_int`.

```
x_pixel, y_pixel, class_int
```

- `x_pixel`: integer, column of the cell centroid **within this tile** in pixels at 0.25 MPP. Origin = tile top-left corner. Range: [0, 1023].
- `y_pixel`: integer, row of the cell centroid **within this tile** in pixels at 0.25 MPP.
- `class_int`: integer class index from Section 4.1.

**Example `<sample_tag>_tile_0042.csv`:**
```
46,7,2
191,100,0
108,191,1
146,173,5
233,117,4
```

**Rules:**
- Cells that fall outside [0, 1023] in either axis must be **excluded** from that tile's CSV.
- If the tile has **zero cells**, still export an empty image file, but **do not create a label CSV** (or create an empty one вЂ” the DataLoader skips tiles not in the filelist anyway).
- Cells at tile boundaries (overlap region): include a cell in a tile if its centroid falls in [tile_x_start, tile_x_start + 1024) and [tile_y_start, tile_y_start + 1024) at 0.25 MPP.

### Coordinate conversion (Xenium Вµm в†’ tile pixel)

```
tile_origin_x_um = tile_col_index * stride_um    # stride_um = 240.0 Вµm (960 px Г— 0.25)
tile_origin_y_um = tile_row_index * stride_um

x_pixel = round((cell_x_um - tile_origin_x_um) / 0.25)
y_pixel = round((cell_y_um - tile_origin_y_um) / 0.25)
```

## 9. Splits Files

`splits/fold_0/train.csv` and `splits/fold_0/val.csv` are plain text files, one
tile stem per line, **no header**:

```
<sample_tag>_tile_0001
<sample_tag>_tile_0002
<sample_tag>_tile_0005
...
```

**Split strategy:**
- Allocate ~80% of tiles to `train.csv`, ~20% to `val.csv`.
- Split by **spatial region** (e.g. top 80% of slide rows в†’ train, bottom 20% в†’ val), not randomly, to avoid tile overlap leakage from the 64 px overlap.
- Keep tiles from different samples together in the same pool (no per-sample split).

## 11. Training Config Template

`${CELLVIT_TRAINING_ROOT}`, materializes the tokenized YAML at
`trainingset/<tissue>/train_configs/<backbone>/.<fold>.resolved.yaml` via
`envsubst`, then invokes
`cellvit/CellViT-plus-plus/cellvit/train/train_cell_classifier.py`.

The authoring template (currently used for pantissue) is:

  hidden_dim: 256

training:
  cache_cell_dataset: true
  batch_size: 64
  epochs: 50

  # Inverse-frequency weights (weight в‰€ 10 / class_percent), capped at 10.
  # Re-derive per tissue from the actual exported-tile class distribution.
  # See trainingset/pantissue/train_configs/SAM-H-x40/fold_0.yaml for the

Notes:
- All paths use `${CELLVIT_TRAINING_ROOT}` (a sibling of the project root) so

- `log_comment` must equal `<tissue>-<task>-<backbone-lower>`; the wrapper uses
  this string to glob for the run directory after training finishes.
- `num_classes` and `label_map` must match `trainingset/<tissue>/label_map.yaml`
  exactly (the same mapping the QuPath export pipeline uses).

## 12. QuPath Groovy Script Requirements

The Groovy script must perform the following steps:

### 12.1 Inputs
- An H&E image opened in the project, with QuST detections already carrying
  pantissue label strings on their PathClass (post `load_mapping.groovy`)
- Output directory root (e.g. `trainingset/pantissue/`) set in the script
- `trainingset/<tissue>/label_map.yaml` (int в†” label-name)
- Split: `"train"` or `"test"`

### 12.2 Export parameters
```
EXPORT_MPP    = 0.25       // Вµm/px
TILE_SIZE_PX  = 1024       // pixels
OVERLAP_PX    = 64         // pixels on each side
STRIDE_PX     = 960        // = TILE_SIZE_PX - OVERLAP_PX
STRIDE_UM     = 240.0      // = STRIDE_PX * EXPORT_MPP
TILE_SIZE_UM  = 256.0      // = TILE_SIZE_PX * EXPORT_MPP
```

### 12.3 Tile export loop (pseudocode)
```
cells = [(roi.centroidX_px * slideMPP, roi.centroidY_px * slideMPP,
          labelToInt[obj.PathClass.name])
         for obj in getDetectionObjects()
         if obj.PathClass != null and obj.PathClass.name in labelToInt]

for row_index in 0..n_rows:
    for col_index in 0..n_cols:
        tile_x_um = col_index * STRIDE_UM
        tile_y_um = row_index * STRIDE_UM

        // Export image region
        region = RegionRequest(tile_x_um, tile_y_um, TILE_SIZE_UM, TILE_SIZE_UM, downsample)
        image = server.readRegion(region)  // rendered as RGB
        save(image, "{output_dir}/{split}/images/{sample_tag}_tile_{idx:04d}.png")

        // Find cells in this tile
        cells_in_tile = [c for c in cells
                         if tile_x_um <= c.x_um < tile_x_um + TILE_SIZE_UM
                         and tile_y_um <= c.y_um < tile_y_um + TILE_SIZE_UM]

        if len(cells_in_tile) < MIN_CELLS: continue  // skip near-empty tiles

        // Write label CSV
        with open("{output_dir}/{split}/labels/{sample_tag}_tile_{idx:04d}.csv", "w") as f:
            for c in cells_in_tile:
                px = round((c.x_um - tile_x_um) / EXPORT_MPP)
                py = round((c.y_um - tile_y_um) / EXPORT_MPP)
                f.write(f"{px},{py},{c.class_int}\n")
```

### 12.4 QuPath downsample factor
QuPath uses `downsample` = `EXPORT_MPP / slide_pixel_size_um`.
Get the slide pixel size from `server.getPixelCalibration().getPixelWidthMicrons()`.

```groovy
double exportMPP = 0.25
double slideMPP  = server.getPixelCalibration().getPixelWidthMicrons()
double downsample = exportMPP / slideMPP
```

### 12.5 Tissue masking (recommended)
Only export tiles that contain tissue. A simple check: skip tiles where the mean
pixel brightness > 240 (mostly white background).

### 12.6 Minimum cell count per tile (recommended)
Skip tiles with fewer than **5 cells** (after filtering to assigned cells).
These contribute little signal and inflate the dataset with near-empty labels.

## 13. Validation Checklist

Before handing the dataset to training, verify:

- [ ] All PNG files are exactly 1024Г—1024 px, RGB, uint8
- [ ] Every PNG in `train/images/` has a matching CSV in `train/labels/` with the same stem
- [ ] All CSV files have no header row, exactly 3 integer columns per row
- [ ] All `x_pixel` and `y_pixel` values are in range [0, 1023]
- [ ] All `class_int` values are in range [0, N-1] (`N` = number of classes in
      `label_map.yaml`; 12 for `pantissue`, 11 for the retired breast scheme)
- [ ] `splits/fold_0/train.csv` and `val.csv` contain only stems that exist in `train/images/`
- [ ] `label_map.yaml` is present at the dataset root
- [ ] No tile stem appears in both train and val
